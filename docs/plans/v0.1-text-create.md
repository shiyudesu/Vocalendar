# v0.1 Text Create

## 1. 版本目标

`v0.1` 的目标是跑通“文本输入 -> 事件草稿 -> 补充缺失字段 -> 用户确认 -> 创建成功 -> 查看最近创建事件”的最小闭环，并通过这个闭环验证自然语言文本创建日程是否具备真实效率价值。

本版本只解决“创建”问题，不提前引入查询、修改、删除、提醒、重复规则和外部同步。

---

## 2. 范围定义

### 2.1 范围内
- 自然语言文本输入
- 将输入转换成结构化事件草稿
- 展示缺失字段、警告信息和补问信息
- 允许用户补充或修正缺失字段
- 用户确认后创建事件
- 展示最近创建事件列表

### 2.2 范围外
- 浏览器录音与语音转写
- 今日 / 明日 / 本周范围查询
- 已创建事件的修改和删除
- 提醒、重复规则、日历同步
- 多用户协作
- `Capacitor` 移动端交互

---

## 3. 用户流程

### 3.1 创建主流程
1. 用户输入自然语言文本
2. `packages/web` 的 Vite React 前端通过配置化 API base URL 将文本请求发送给 `packages/api` 草稿创建接口
3. 服务端返回结构化 `draft`
4. 如果 `draft.canSave = false`，前端引导用户补充缺失字段
5. 用户确认草稿内容
6. 前端请求事件创建接口
7. 服务端返回创建成功的 `event`
8. 前端刷新最近创建事件列表

### 3.2 缺失字段补充流程
1. 草稿返回 `missingFields`
2. 用户通过补充自然语言或直接填写字段修正草稿
3. 前端请求草稿更新接口
4. 服务端重新计算 `parsed`、`missingFields`、`warnings` 和 `canSave`
5. 直到 `canSave = true` 后，允许进入创建

### 3.3 当前版本实现边界
- `v0.1` 采用前后端包分离实现：`packages/web` 提供 Vite + React 浏览器入口，`packages/api` 提供 Hono HTTP API
- 仓库采用 `pnpm workspace` + Turborepo 管理 `packages/web`、`packages/api` 与后续共享 schema/type
- 本版本只交付 `Web` 入口；`Capacitor` 移动端壳体不在范围内，但接口契约和前端交互必须可复用

## 4. 版本内 API 决策

本节是 `v0.1` 具体 API 的先行来源；稳定后同步到 `docs/api-spec.md`。

### 4.1 资源命名
- 草稿资源：`drafts`
- 事件资源：`events`

`v0.1` 不使用 `/parse`、`/clarify` 这类动作路径，统一通过资源创建和资源更新表达。

### 4.2 `POST /api/v1/drafts`

用途：
- 根据自然语言文本创建事件草稿

请求体：

```json
{
  "sourceText": "明天下午三点和张总在国贸喝咖啡",
  "timezone": "Asia/Shanghai",
  "referenceAt": "2026-05-29T02:00:00Z",
  "source": "text"
}
```

成功返回：
- `draft`

失败场景：
- 文本为空
- 时区非法
- `referenceAt` 非法或缺失
- 完全无法解析出有效草稿

### 4.3 `PATCH /api/v1/drafts/:draftId`

用途：
- 补充缺失字段或修正草稿

请求体：

```json
{
  "userInput": "下午三点开始",
  "referenceAt": "2026-05-29T02:05:00Z",
  "fields": {
    "location": "国贸"
  }
}
```

规则：
- `userInput` 和 `fields` 至少提供一个
- 提供 `userInput` 时，必须同时提供 `referenceAt`
- `fields` 只允许更新 `title`、`startAt`、`endAt`、`timezone`、`location`、`participants`
- 更新后必须重新计算 `missingFields`、`warnings` 和 `canSave`

### 4.4 `POST /api/v1/events`

用途：
- 将一个可保存草稿创建为正式事件

请求体：

```json
{
  "draftId": "drf_123"
}
```

规则：
- `draftId` 必填
- 草稿必须存在且 `canSave = true`

### 4.5 `GET /api/v1/events`

用途：
- 返回最近创建的事件

查询参数：
- `mode=recent`
- `limit`，默认 `5`，最大 `10`

`v0.1` 不支持范围查询；范围查询属于 `v0.2 Text Query` 决策范围。

## 5. 核心对象与数据规则

### 5.1 EventDraft

关键字段：
- `draftId`
- `sourceText`
- `source`
- `referenceAt`
- `normalizedText`
- `parsed`
- `missingFields`
- `warnings`
- `canSave`
- `clarificationPrompt`

`parsed` 在 `v0.1` 至少覆盖：
- `title`
- `startAt`
- `endAt`
- `timezone`
- `location`
- `participants`

### 5.2 Event

保存后的事件在 `v0.1` 至少包含：
- `id`
- `title`
- `startAt`
- `endAt`
- `timezone`
- `location`
- `participants`
- `source`
- `status`
- `createdAt`
- `updatedAt`

### 5.3 保存前最低要求
- `title` 非空
- `startAt` 非空
- `timezone` 非空
- 可追踪输入来源

### 5.4 文本解析策略

#### 时间解析技术
- `v0.1` 在 Node/TypeScript 运行时内使用 `chrono-node` 作为基础时间解析库
- 服务层基于 `chrono-node` 输出做二次归一化，至少覆盖阿拉伯数字标准化、中文时段词映射和业务时区换算
- 当前版本不引入独立 Duckling 服务，避免额外部署复杂度；也不把 LLM 作为主解析器，避免在基础 CRUD 阶段引入不可控的结构化漂移

#### 意图与实体识别策略
- `v0.1` 不做独立“意图识别”；`POST /drafts` 默认表示创建事件草稿
- 解析顺序为：先抽取时间表达，再从剩余文本中用规则提取 `participants`、`location`，最后生成 `title` 候选
- 实体抽取工具采用纯 TypeScript 规则引擎，不引入通用中文 NLP 库；实现方式为 `normalize -> candidate spans -> consume spans -> residual title`
- `participants` 优先识别“和/跟/约/见”等连接词后的称谓或人名片段
- `location` 优先识别“在/到/去/于”等介词后的地点片段
- 触发词本身不属于实体值，例如“和小王在星巴克”中 `participants=["小王"]`、`location="星巴克"`
- 实体片段在遇到下一个触发词、动作词、时间词、标点或句尾时截断
- `title` 通过剩余文本生成：先从原句中扣除时间、地点、参与人 span，再清理孤立触发词和多余空白；如果仍能形成稳定动作短语，则作为 `title`，否则进入补问流程
- 当前版本不做分词、词性分析或通用 NER；只处理和事件创建直接相关的轻量模式

#### 重叠消解与优先级
- span 消解按固定顺序执行：`time -> location -> participants -> title`
- 优先级规则为：`time` 高于其他实体；同级内“带明确触发词的候选”优先于“无触发词的弱候选”
- 已被高优先级 span 占用的字符区间，不允许再次分配给其他实体
- 例如“明天下午三点在国贸约张总喝咖啡”会依次解析为 `time=明天下午三点`、`location=国贸`、`participants=["张总"]`、`title=喝咖啡`

#### 规则失效时的兜底
- `participants` 和 `location` 采用“有证据才填”的保守策略；未识别时分别返回空数组或 `null`
- 当前版本不做没有触发词的强猜测，也不依赖联系人词典或地点词典做二次推断
- `title` 和 `startAt` 属于关键字段；如果规则无法稳定生成，就通过 `missingFields` 和 `clarificationPrompt` 进入补问
- 后续版本可以评估基于联系人上下文、常用地点词典或 LLM 的增强识别，但不属于 `v0.1`

#### 相对时间基准
- 所有相对时间表达都基于请求中的 `referenceAt` 和 `timezone` 共同解释
- `referenceAt` 由前端在用户点击提交时生成，使用 UTC ISO 8601 字符串
- 服务端必须先将 `referenceAt` 转换到 `timezone` 对应的本地时间，再解释“明天”“下周五”“今晚”等相对表达
- 不允许直接使用服务器收到请求的系统时间替代用户提交时刻，除非内部调试工具明确走降级路径

#### 中文时间歧义处理
- 包含明确小时的表达可以直接归一化，例如“下午三点” -> `15:00`
- 只包含日期、不包含具体钟点的表达，不自动猜测 `startAt`，例如“明天”“下周五下午”
- 对上述歧义表达，服务端返回部分草稿，并通过 `missingFields=["startAt"]` 和 `clarificationPrompt` 追问具体开始时间
- `endAt` 在 `v0.1` 仍然允许为空；除非文本中明确给出结束时间，否则不默认补全

#### 解析失败与容错策略
- 采用“尽力提取，但不臆造必填字段”的保守策略
- 只要文本里还能识别出任何有用事件信号，例如日期、时间、地点、参与人或标题候选之一，就返回部分 `draft`
- 只有在文本完全无法形成可操作草稿时，才返回 `DRAFT_PARSE_FAILED`
- 解析出的不确定信息应进入 `warnings` 或 `clarificationPrompt`，而不是静默写入最终事件

## 6. 工作拆分（PR 粒度）

按“每个 PR 只做一件事”原则，`v0.1` 当前以“主干已存在实现”为新基线重新拆分如下。每个 PR 须包含清晰的标题、功能描述、实现思路和测试方式。

### PR A：共享 Schema、Monorepo 与 API 骨架基线

- **功能描述**：建立 pnpm workspace、共享 Zod Schema、Hono API 骨架与基础测试框架。
- **实现思路**：在 `packages/schemas` 定义 `v0.1` 核心 schema；在 `packages/api` 接入 Hono 服务与统一响应结构；打通 `packages/web` / `packages/api` 对共享包的导入链路。
- **测试方式**：`pnpm check` 通过；共享包可被 Web 和 API 导入；基础 API 测试可运行。
- **当前状态**：已完成，作为 `v0.1` 当前主干的新基线。

### PR B：API — 草稿创建接口（POST /api/v1/drafts）

- **功能描述**：根据自然语言文本创建事件草稿，返回结构化 `draft`、`missingFields`、`warnings` 和 `clarificationPrompt`。
- **实现思路**：在 Hono route 中完成 Zod 校验和错误映射；服务层接入 `chrono-node` 做基础时间解析；基于规则引擎抽取 `location`、`participants` 和 `title`；按第 5 节优先级与兜底策略处理歧义和失败。
- **测试方式**：覆盖成功路径、仅日期不含具体钟点时返回部分 draft、文本为空或时区非法等校验失败、完全无法解析时返回 `DRAFT_PARSE_FAILED`。
- **当前状态**：未完成，是当前最核心缺口。

### PR C：API — 草稿更新、事件创建与最近列表闭环

- **功能描述**：实现 `PATCH /api/v1/drafts/:draftId`，并收口 `POST /api/v1/events`、`GET /api/v1/events` 为一组可用闭环接口。
- **实现思路**：通过 Hono routes 实现对应 `/api/v1/*` 端点；草稿更新时复用解析逻辑重新计算 `parsed`、`missingFields` 和 `canSave`；事件创建时校验 `canSave = true`；最近事件列表按 `createdAt` 倒序返回。
- **测试方式**：补充字段后 draft 可保存、创建事件成功、查询最近事件列表、草稿不可保存时创建失败。
- **当前状态**：schema、部分 route 和部分测试已提前落地；`PATCH /drafts/:draftId` 仍是占位实现，需要统一收口。

### PR D：Web — 文本输入与草稿展示

- **功能描述**：提供自然语言文本输入入口，调用草稿创建 API，展示返回的 `draft`、缺失字段、警告和失败状态。
- **实现思路**：Vite React SPA 组件、文本输入框、API Client 封装、草稿卡片展示组件。
- **测试方式**：端到端验证从输入文本到草稿展示的完整流程，包括解析成功和部分解析的展示态。
- **当前状态**：尚未开始，当前只有占位 landing 页面。

### PR E：Web — 草稿补充、确认创建与最近列表

- **功能描述**：允许用户补充缺失字段或直接修改草稿字段，确认后创建正式事件并刷新最近列表。
- **实现思路**：补充输入框、字段编辑表单、调用 `PATCH /drafts` 和 `POST /events`、创建成功后刷新 `GET /events`。
- **测试方式**：验证缺失字段补充后 draft 变为可保存、确认创建后事件出现在最近列表，并输出最小可执行的手动验证清单或 e2e 覆盖。
- **当前状态**：尚未开始。

## 7. 验收标准

- 用户可以通过一条自然语言文本生成事件草稿
- 当草稿缺少必要字段时，系统能明确指出 `missingFields`
- 用户补充缺失字段后，系统可以重新生成可保存草稿
- 用户确认后可以成功创建事件
- 创建成功后可以看到最近创建事件列表
- 失败场景可以区分校验失败、解析失败和保存失败
- 每次合并后主分支代码保持可运行状态，可随时演示复现

---

## 8. 验证方式

至少覆盖以下验证：

- 草稿创建成功路径
- 草稿创建失败路径
- 草稿补充缺失字段后转为可保存
- 创建事件成功路径
- 创建事件时草稿缺失字段失败路径
- 最近创建事件列表展示路径

如果自动化脚本尚未建立，至少补一份可执行的手动验证清单。

## 9. 风险与非目标

### 9.1 主要风险
- 自然语言解析不稳定，导致草稿可用率低
- 缺失字段补问过长，破坏创建效率
- 草稿和正式事件字段语义不一致，导致确认后仍然保存错误

### 9.2 非目标
- 不验证复杂日程查询
- 不验证编辑、删除、提醒和同步能力
- 不为后续版本提前锁死接口形态
