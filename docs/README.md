# 文档索引

## 1. 目的

`docs/` 目录用于管理 Vocalendar 的产品、技术、接口和版本计划文档。每份文档都有明确边界，避免同一信息在多个文件里重复维护。

## 2. 文档边界

| 文档 | 单一职责 | 应包含 | 不应包含 |
| --- | --- | --- | --- |
| `docs/prd.md` | 产品需求与版本路线 | 产品目标、用户问题、核心能力、版本划分、长期指标 | 技术细节、接口字段、开发任务分解 |
| `docs/architecture.md` | 技术架构与工程规范 | 技术栈、系统模块、目录结构、编码底线、工程约束 | 产品路线、逐版本排期、接口示例的大量业务细节 |
| `docs/api-spec.md` | API 通用规范与当前稳定契约 | 通用规则、响应格式、错误码、命名规范、已进入开发版本的稳定接口契约 | 未来版本的预写接口、产品愿景、工程管理规则 |
| `docs/plans/*.md` | 单版本开发计划 | 目标、范围、用户流程、工作拆分、验收标准、风险、该版本接口决策 | 长期产品路线、全局技术总则 |
| `AGENTS.md` | 仓库级协作规则 | AI/开发者工作方式、文档同步要求、最低验证要求 | 产品需求细节、API 字段清单 |

## 3. 建议阅读顺序

### 3.1 新成员或新 Agent
1. `AGENTS.md`
2. `docs/README.md`
3. `docs/prd.md`
4. 如果当前已有活跃版本 plan，读取对应的 `docs/plans/*.md`
5. `docs/architecture.md`
6. `docs/api-spec.md`

### 3.2 版本规划
1. 先更新 `docs/prd.md` 中的版本目标或边界
2. 在开始该版本开发前，创建或更新对应 `docs/plans/*.md`，先确定该版本的具体 API
3. 当该版本进入开发且接口契约稳定后，再同步更新 `docs/api-spec.md`
4. 如果涉及技术结构或工程约束变化，更新 `docs/architecture.md`

### 3.3 开发实现
1. 读取活跃版本的计划文档
2. 依据 `architecture.md` 保持结构与代码约束一致
3. 先按当前版本 plan 落实版本内接口，再按 `api-spec.md` 对齐通用规范与当前稳定契约
4. 如实现结果改变了当前版本接口，先更新 plan，再同步 `api-spec.md`

## 4. 目录结构

```text
docs/
  README.md
  prd.md
  architecture.md
  api-spec.md
  plans/
    # 在进入某个版本开发前，按约定命名创建对应 plan
```

## 5. 更新原则

- 产品边界变化：更新 `prd.md`
- 技术方案或工程底线变化：更新 `architecture.md`
- 当前稳定接口契约或通用 API 规范变化：更新 `api-spec.md`
- 单版本范围、流程、验收或版本内接口决策变化：更新对应 `plans/*.md`
- 如果某版本尚未开始开发，不提前产出该版本 plan
- 协作方式或 Agent 规则变化：更新 `AGENTS.md`

如果一个改动同时影响多份文档，必须在同一个变更中同步完成，避免文档漂移。
