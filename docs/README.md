# 文档索引

## 1. 目的

`docs/` 目录用于管理 Vocalendar 的产品、技术、接口和模块计划文档。每份文档都有明确边界，避免同一信息在多个文件里重复维护。

## 2. 文档边界

| 文档 | 单一职责 | 应包含 | 不应包含 |
| --- | --- | --- | --- |
| `docs/prd.md` | 产品需求与版本路线 | 产品目标、用户问题、核心能力、版本划分、长期指标 | 技术细节、接口字段、开发任务分解 |
| `docs/architecture.md` | 技术架构与工程规范 | 技术栈、系统模块、目录结构、编码底线、工程约束 | 产品路线、逐版本排期、接口示例的大量业务细节 |
| `docs/api-spec.md` | API 通用规范与全量预定义契约 | 通用规则、响应格式、错误码、命名规范、当前与预留接口契约 | 产品愿景、工程管理规则 |
| `docs/modules/*.md` | 模块开发计划 | 模块定位、接口契约、PR 拆分、验收标准、风险、当前状态 | 长期产品路线、全局技术总则 |
| `AGENTS.md` | 仓库级协作规则 | AI/开发者工作方式、文档同步要求、最低验证要求 | 产品需求细节、API 字段清单 |

## 3. 建议阅读顺序

### 3.1 新成员或新 Agent
1. `AGENTS.md`
2. `docs/README.md`
3. `docs/prd.md`
4. 读取当前活跃模块对应的 `docs/modules/*.md`
5. `docs/architecture.md`
6. `docs/api-spec.md`

### 3.2 模块规划
1. 先更新 `docs/prd.md` 中的模块总览
2. 在开始模块开发前，创建或更新对应 `docs/modules/*.md`，先确定模块的接口契约
3. 当模块接口契约需要新增或变更时，同步更新 `docs/api-spec.md`
4. 如果涉及技术结构或工程约束变化，更新 `docs/architecture.md`

### 3.3 开发实现
1. 读取活跃模块的计划文档
2. 依据 `architecture.md` 保持结构与代码约束一致
3. 先按当前模块 plan 落实模块内接口，再按 `api-spec.md` 对齐通用规范与全量预定义契约
4. 如实现结果改变了模块接口，先更新模块 plan，再同步 `api-spec.md`

## 4. 目录结构

```text
docs/
  README.md
  prd.md
  architecture.md
  api-spec.md
  modules/
    # 各模块开发计划，按模块名命名创建
```

## 5. 更新原则

- 产品边界变化：更新 `prd.md`
- 技术方案或工程底线变化：更新 `architecture.md`
- API 通用规范或全量预定义契约变化：更新 `api-spec.md`
- 单模块范围、流程、验收或模块内接口决策变化：更新对应 `modules/*.md`
- 如果某模块尚未开始开发，不提前产出该模块 plan
- 协作方式或 Agent 规则变化：更新 `AGENTS.md`

如果一个改动同时影响多份文档，必须在同一个变更中同步完成，避免文档漂移。
