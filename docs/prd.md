# Vocalendar 产品需求文档（PRD）

## 1. 文档定位

本文件是 Vocalendar 的长期产品主文档，只负责回答四类问题：
- 产品要解决什么问题
- 面向哪些用户与场景
- 长期核心能力是什么
- 各开发版本如何划分与递进

本文件不负责技术实现、接口细节、目录结构、编码规范，也不展开某个版本的具体开发步骤。这些内容分别由 `architecture.md`、`api-spec.md` 和 `docs/plans/*.md` 承担。

---

## 2. 产品概述

### 2.1 产品愿景
Vocalendar 希望成为一个“自然语言优先、逐步演进到语音优先”的个人日程管理入口，让用户用一句自然语言快速完成日程记录、确认与查询，并在能力成熟后自然切换到语音输入。

### 2.2 当前产品策略
- 首发形态为 `Web`
- 先验证 `自然语言驱动的日程 CRUD 闭环` 是否具备真实效率价值
- 在 Web 中先稳定草稿、确认和事件模型，再在后续版本接入语音输入
- 在 Web 中验证完成的交互、数据模型和语义能力，后续通过 `Capacitor` 渐进承载移动端

### 2.3 长期价值主张
- `更快`：减少传统表单式日程创建步骤
- `更顺`：理解自然语言中的时间、地点、人物与提醒表达
- `更稳`：输出结构化、可确认、可修改、可追溯的事件结果

## 3. 用户与核心问题

### 3.1 目标用户

| 用户类型 | 典型特征 | 主要需求 |
| --- | --- | --- |
| 知识工作者 | 会议多、切换频繁 | 快速记录临时安排 |
| 多任务场景用户 | 双手被占用、注意力分散 | 在不中断当前任务的情况下记日程 |
| 效率工具用户 | 已习惯日历、提醒类产品 | 用更自然的方式输入结构化事件 |
| 轻度无障碍需求用户 | 打字效率低或依赖语音辅助 | 更低门槛的日程录入方式 |

### 3.2 关键用户任务
- 当我突然想到一条安排时，我希望立刻说一句话把它记下来
- 当我的表达不完整时，我希望系统补问关键字段而不是直接失败
- 当系统准备保存一条事件时，我希望先看到或听到明确确认
- 当我想确认刚刚记录的内容时，我希望快速查询到结果

### 3.3 当前替代方案的不足
- 传统日历创建流程依赖表单填写，步骤多、打断强
- 通用语音助手不一定稳定产出结构化日程
- 模糊时间、提醒、重复规则等表达容易被误解
- 用户往往不确定系统是否真的“记对了”或“保存了”

---

## 4. 产品原则

1. `自然语言优先`
   核心流程优先围绕一句自然语言生成结构化事件；早期版本先用文本输入稳定闭环，再逐步接入语音。
2. `结构化优先`
   系统输出必须是事件草稿或事件记录，而不是仅返回自然语言回答。
3. `确认优先`
   对关键字段存在歧义时，优先补问或确认，避免静默写错。
4. `版本收敛`
   每个版本只验证少数关键假设，避免范围失控。
5. `跨端可迁移`
   Web 阶段沉淀的事件模型、接口契约和核心交互流程应可复用于 `Capacitor` 移动端。

## 5. 长期需求总览

### 5.1 核心能力地图

| 能力模块 | 说明 | 长期优先级 |
| --- | --- | --- |
| 语音输入与转写 | 获取用户语音并转成文本 | P0 |
| 语义理解与时间解析 | 提取事件主体、时间、地点、提醒等字段 | P0 |
| 事件草稿与确认 | 生成结构化草稿并与用户确认 | P0 |
| 事件管理 | 创建、查询、修改、删除事件 | P0-P1 |
| 提醒管理 | 设置和触发提醒 | P1 |
| 重复事件 | 支持周期性规则 | P1 |
| 日历同步 | 与外部日历互通 | P2 |
| 偏好学习 | 默认时长、常用地点、常见联系人等 | P2 |
| 多端承载 | Web、`Capacitor` 移动端及潜在其他终端 | P2 |

### 5.2 长期需求边界
当前产品长期围绕“个人日程管理”构建，不以以下方向为优先：
- 团队协作审批与复杂排班
- 邮件邀请解析与会议室预订
- 完整待办或项目管理替代
- 智能音箱、车机、可穿戴设备的首发支持

## 6. 版本路线图

### 6.1 版本划分原则
- 每个版本必须有清晰目标、范围边界和验收口径
- 每个版本都要对应一个独立的 `docs/plans/*.md` 计划文档
- 后续版本建立在前一版本验证结论之上，不跳跃开发

### 6.2 版本总览

| 版本 | 阶段目标 | 核心验证点 | 计划文档命名约定 |
| --- | --- | --- | --- |
| `v0.1 Text Create` | 跑通文本创建最小闭环 | 用户能否通过一句自然语言文本创建事件 | `docs/plans/v0.1-text-create.md` |
| `v0.2 Text Query` | 补齐基础查询能力 | 用户能否在创建后快速回查今天、明天和本周事件 | `docs/plans/v0.2-text-query.md` |
| `v0.3 Text Update` | 支持修改已创建事件 | 用户能否自然地修改时间、标题和地点等关键字段 | `docs/plans/v0.3-text-update.md` |
| `v0.4 Text Delete` | 支持删除已创建事件 | 用户能否安全、明确地删除错误或过期事件 | `docs/plans/v0.4-text-delete.md` |
| `v0.5 Voice Input` | 接入语音输入与转写能力 | 用户能否通过语音进入既有自然语言流程 | `docs/plans/v0.5-voice-input.md` |
| `v0.6 Reminder & Recurrence` | 支持提醒和基础重复规则 | 用户能否管理更接近日常使用的提醒与周期性事件 | `docs/plans/v0.6-reminder-recurrence.md` |
| `v0.7 Calendar Sync` | 支持外部日历同步 | 用户能否把 Vocalendar 与现有日历体系连接起来 | `docs/plans/v0.7-calendar-sync.md` |
| `v1.0 Web Complete` | 形成完整可用的 Web 产品 | 账号、数据、同步和稳定性是否支撑正式使用 | `docs/plans/v1.0-web-complete.md` |
| `v2.0 Mobile via Capacitor` | 在移动端建立主要使用场景 | Web 验证过的能力能否通过 `Capacitor` 在移动端放大价值 | `docs/plans/v2.0-mobile-via-capacitor.md` |

### 6.3 各版本定义

#### v0.1 Text Create
目标：
- 跑通“文本输入 -> 事件草稿 -> 用户确认 -> 创建成功”的最小闭环
- 识别高频自然语言文本表达模式和主要失败原因

范围内：
- 文本输入
- 事件主体、日期、开始时间的基础解析
- 结构化确认
- 最近创建事件展示

范围外：
- 浏览器录音与语音转写
- 外部日历同步
- 提醒推送
- 重复事件
- 多用户协作
- `Capacitor` 移动端支持

#### v0.2 Text Query
目标：
- 在完成文本创建后，补齐最基础的自然语言查询能力
- 验证用户是否会把查询作为创建后的高频辅助动作

范围内：
- 今日/明日/本周基础查询
- 最近创建事件的快速回查
- 查询结果的基础展示与摘要
- 为查询场景补齐必要埋点

#### v0.3 Text Update
目标：
- 让用户可以修改已创建但信息不准确的事件
- 优先验证“改时间”是否是最常见、最有价值的修改操作

范围内：
- 修改事件开始时间和结束时间
- 修改事件标题
- 修改事件地点
- 基础修改确认与结果反馈

#### v0.4 Text Delete
目标：
- 让用户可以删除错误创建、临时取消或过期的事件
- 保证删除流程安全可理解，不因误删破坏信任感

范围内：
- 单条事件删除
- 删除前确认
- 删除后的结果反馈
- 基础删除日志或撤销策略预留

#### v0.5 Voice Input
目标：
- 在现有文本 CRUD 基础上接入语音输入和转写能力
- 验证语音是否是比文本更高频、更高效的输入入口

范围内：
- 浏览器录音或受控语音输入
- 转写结果展示与纠错
- 语音转写文本复用既有 drafts / events 流程
- 录音权限、失败重试和必要埋点

#### v0.6 Reminder & Recurrence
目标：
- 让事件具备更接近真实日常使用的提醒和周期能力
- 验证提醒与重复规则是否显著提升留存和复用价值

范围内：
- 提醒时间设置
- 提醒规则查看与修改
- 每天 / 每周 / 每月等基础重复事件
- 重复规则的基础展示

#### v0.7 Calendar Sync
目标：
- 让 Vocalendar 能与现有日历工具协同工作
- 降低用户迁移成本，避免形成新的信息孤岛

范围内：
- 外部日历账号连接
- 基础导入或同步能力
- 同步状态反馈
- 基础冲突提示

#### v1.0 Web Complete
目标：
- 形成完整、稳定、可持续使用的 Web 产品版本

范围内：
- 账号体系与数据隔离
- 可靠数据存储
- 监控、日志、隐私说明和恢复机制
- 查询、修改、删除、提醒、重复和同步能力的统一收口

#### v2.0 Mobile via Capacitor
目标：
- 基于 Web 已验证的能力，通过 `Capacitor` 在移动端提供更自然、更高频的语音日程体验

范围内：
- `Capacitor` 壳体与移动端发布能力
- 受移动端权限控制的语音交互
- 推送提醒与系统级集成
- 更适合碎片场景的移动端交互

## 7. 当前开发基线

当前默认开发基线为 `v0.1 Text Create`。

当前版本必须验证三件事：
- 用户能否在 Web 中顺利完成一次文本创建
- 系统能否把高频自然语言解析成可确认的事件草稿
- 补问与确认流程是否足够短，且不会明显打断用户

当前版本最低交付结果：
- 可以输入一条自然语言文本
- 可以展示解析结果
- 可以补充缺失字段
- 可以在确认后保存事件
- 可以查看最近创建的事件

## 8. 成功指标

### 8.1 长期指标
- 自然语言创建完成率
- 语音输入使用率
- 首次创建成功率
- 平均创建耗时
- 平均补问轮次
- 周活跃创建用户数
- 次周留存

### 8.2 当前版本关注指标
- 文本创建闭环率
- 解析可用率
- 缺失字段分布
- 用户纠错频率

指标定义与采集方式不在本文件展开，由对应版本计划文档补充。

## 9. 文档协同规则

- `docs/prd.md` 只维护产品方向、长期需求与版本路线
- `docs/architecture.md` 维护技术栈、目录规范和编码底线
- `docs/api-spec.md` 维护 API 通用规范，以及已进入开发版本的当前稳定接口契约和核心数据对象
- `docs/plans/*.md` 在进入对应版本开发前按需创建，维护该版本的具体开发计划、验收标准、发布节奏与版本内接口决策
- 仓库级协作规则由根目录 `AGENTS.md` 维护

当版本目标、产品边界或路线调整时，先更新 PRD；当某个版本的具体实施或版本内 API 决策变化时，更新对应计划文档，再在契约稳定后同步 `api-spec.md`。
