# Vocalendar 产品需求文档（PRD）

## 1. 文档定位

本文件是 Vocalendar 的长期产品主文档，只负责回答四类问题：
- 产品要解决什么问题
- 面向哪些用户与场景
- 长期核心能力是什么
- 各开发版本如何划分与递进

本文件不负责技术实现、接口细节、目录结构、编码规范，也不展开某个版本的具体开发步骤。这些内容分别由 `architecture.md`、`api-spec.md` 和 `docs/modules/*.md` 承担。

---

## 2. 产品概述

### 2.1 产品愿景
Vocalendar 希望成为一个“自然语言优先、逐步演进到语音优先”的个人日程管理入口，让用户用一句自然语言快速完成日程记录、确认与查询，并在能力成熟后自然切换到语音输入。

### 2.2 当前产品策略
- 首发形态为 `Web`
- 先验证 `自然语言驱动的日程 CRUD 闭环` 是否具备真实效率价值
- 在 Web 中先稳定草稿、确认和事件模型，再在后续版本接入语音输入
- 在 Web 中验证完成的交互、数据模型和语义能力，后续通过 `Capacitor` 渐进承载移动端

### 2.3 长期价值主张
- `更快`：减少传统表单式日程创建步骤
- `更顺`：理解自然语言中的时间、地点、人物与提醒表达
- `更稳`：输出结构化、可确认、可修改、可追溯的事件结果

## 3. 用户与核心问题

### 3.1 目标用户

| 用户类型 | 典型特征 | 主要需求 |
| --- | --- | --- |
| 知识工作者 | 会议多、切换频繁 | 快速记录临时安排 |
| 多任务场景用户 | 双手被占用、注意力分散 | 在不中断当前任务的情况下记日程 |
| 效率工具用户 | 已习惯日历、提醒类产品 | 用更自然的方式输入结构化事件 |
| 轻度无障碍需求用户 | 打字效率低或依赖语音辅助 | 更低门槛的日程录入方式 |

### 3.2 关键用户任务
- 当我突然想到一条安排时，我希望立刻说一句话把它记下来
- 当我的表达不完整时，我希望系统补问关键字段而不是直接失败
- 当系统准备保存一条事件时，我希望先看到或听到明确确认
- 当我想确认刚刚记录的内容时，我希望快速查询到结果

### 3.3 当前替代方案的不足
- 传统日历创建流程依赖表单填写，步骤多、打断强
- 通用语音助手不一定稳定产出结构化日程
- 模糊时间、提醒、重复规则等表达容易被误解
- 用户往往不确定系统是否真的“记对了”或“保存了”

---

## 4. 产品原则

1. `自然语言优先`
   核心流程优先围绕一句自然语言生成结构化事件；早期版本先用文本输入稳定闭环，再逐步接入语音。
2. `结构化优先`
   系统输出必须是事件草稿或事件记录，而不是仅返回自然语言回答。
3. `确认优先`
   对关键字段存在歧义时，优先补问或确认，避免静默写错。
4. `版本收敛`
   每个版本只验证少数关键假设，避免范围失控。
5. `跨端可迁移`
   Web 阶段沉淀的事件模型、接口契约和核心交互流程应可复用于 `Capacitor` 移动端。

## 5. 长期需求总览

### 5.1 核心能力地图

| 能力模块 | 说明 | 长期优先级 |
| --- | --- | --- |
| 语音输入与转写 | 获取用户语音并转成文本 | P0 |
| 语义理解与时间解析 | 提取事件主体、时间、地点、提醒等字段 | P0 |
| 事件草稿与确认 | 生成结构化草稿并与用户确认 | P0 |
| 事件管理 | 创建、查询、修改、删除事件 | P0-P1 |
| 提醒管理 | 设置和触发提醒 | P1 |
| 重复事件 | 支持周期性规则 | P1 |
| 日历同步 | 与外部日历互通 | P2 |
| 偏好学习 | 默认时长、常用地点、常见联系人等 | P2 |
| 多端承载 | Web、`Capacitor` 移动端及潜在其他终端 | P2 |

### 5.2 长期需求边界
当前产品长期围绕“个人日程管理”构建，不以以下方向为优先：
- 团队协作审批与复杂排班
- 邮件邀请解析与会议室预订
- 完整待办或项目管理替代
- 智能音箱、车机、可穿戴设备的首发支持

## 6. 版本路线图

### 6.1 模块划分原则
- 每个模块必须有清晰职责边界、接口契约和验收标准
- 每个模块对应一个独立的 `docs/modules/*.md` 计划文档
- 模块间通过 `docs/api-spec.md` 预先定义的接口契约解耦
- 模块内部按功能点拆分为独立 PR，模块间允许并行推进

### 6.2 模块总览

本项目采用「接口契约先行、模块独立演进」策略：

1. **接口先行**：所有模块间契约在 `docs/api-spec.md` 中预先定义，模块开发前必须完成接口评审
2. **模块自治**：各模块有独立开发计划、独立验收标准、独立可运行状态
3. **并行推进**：模块间仅通过接口契约耦合，无其他依赖时允许并行开发

| 模块 | 职责 | 当前状态 | 计划文档 |
|------|------|---------|--------|
| **Schemas** | 全量 schema、类型、错误码 | 进行中 | `docs/modules/schemas.md` |
| **Parser** | 自然语言解析引擎（时间 + 实体抽取） | 待开始 | `docs/modules/parser.md` |
| **API** | Hono HTTP 服务、路由、数据库、业务编排 | 待开始 | `docs/modules/api.md` |
| **Web** | Vite React 浏览器入口、用户交互 | 待开始 | `docs/modules/web.md` |
| **Voice** | 语音输入、录音、转写链路 | 待开始 | `docs/modules/voice.md` |
| **Mobile** | Capacitor 移动端壳体、原生集成 | 待开始 | `docs/modules/mobile.md` |

### 6.3 模块详细路线

各模块的具体实现目标、范围边界、开发任务和验收标准，已拆分到独立模块文档中：

| 模块 | 文档 | 当前重点 |
|------|------|---------|
| Schemas | [`docs/modules/schemas.md`](modules/schemas.md) | schema 补全（Query/Reminder） |
| Parser | [`docs/modules/parser.md`](modules/parser.md) | 范围查询解析、修改意图识别 |
| API | [`docs/modules/api.md`](modules/api.md) | 范围查询、修改删除、Reminder |
| Web | [`docs/modules/web.md`](modules/web.md) | 查询页面、编辑删除、PR E 收尾 |
| Voice | [`docs/modules/voice.md`](modules/voice.md) | 录音、ASR、转写纠错 |
| Mobile | [`docs/modules/mobile.md`](modules/mobile.md) | Capacitor 壳体、推送 |

PRD 不再维护逐版本实现细节，模块文档是唯一的实施来源。

## 7. 当前开发基线

当前开发基线按模块划分：

| 模块 | 当前状态 | 验证目标 |
|------|---------|---------|
| Schemas | Draft/Event 基础 schema 已定义 | 前后端可共享同一套类型 |
| Parser | 基础规则已实现 | 高频句式能否解析出可确认的草稿 |
| API | drafts/events CR 已实现 | 接口能否闭环（创建→确认→保存→列表） |
| Web | 输入与草稿展示已实现 | 用户能否在浏览器完成一次文本草稿生成 |

当前最低交付结果：
- 可以输入一条自然语言文本
- 可以展示解析结果
- 可以查看解析结果中的缺失字段、警告和补问提示

## 8. 成功指标

### 8.1 长期指标
- 自然语言创建完成率
- 语音输入使用率
- 首次创建成功率
- 平均创建耗时
- 平均补问轮次
- 周活跃创建用户数
- 次周留存

### 8.2 当前模块关注指标
- 文本创建闭环率
- 解析可用率
- 缺失字段分布
- 用户纠错频率

指标定义与采集方式不在本文件展开，由对应模块计划文档补充。

---

## 9. 文档协同规则

- `docs/prd.md` 只维护产品方向、长期需求与模块路线
- `docs/architecture.md` 维护技术栈、目录规范和编码底线
- `docs/api-spec.md` 维护全量预定义接口契约，是模块开发的共同基线
- `docs/modules/*.md` 按模块维护具体开发计划、验收标准与模块内接口决策
- `docs/plans/` 保留历史版本文档，不再新增
- 仓库级协作规则由根目录 `AGENTS.md` 维护

当产品边界或模块路线调整时，先更新 PRD；当某个模块的具体实施或模块内 API 决策变化时，更新对应模块计划文档，再在契约稳定后同步 `api-spec.md`。
