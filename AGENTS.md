# AGENTS.md

## 1. 作用

本文件用于约束 AI Agent 和开发者在 Vocalendar 仓库中的协作方式。它不重复产品需求和接口细节，只定义工作流、文档同步规则和最低工程要求。

## 2. 必读顺序

开始任何非微小改动前，按以下顺序读取：
1. `docs/README.md`
2. `docs/prd.md`
3. 当前活跃版本对应的 `docs/plans/*.md`
4. `docs/architecture.md`
5. `docs/api-spec.md`

如果任务只改接口契约或架构，也至少要先读 `docs/README.md` 以确认文档边界。

## 3. 文档单一事实来源

| 主题 | 单一事实来源 |
| --- | --- |
| 产品目标、范围、版本路线 | `docs/prd.md` |
| 技术栈、目录规范、编码底线 | `docs/architecture.md` |
| API 通用规范、当前稳定接口契约与核心对象 | `docs/api-spec.md` |
| 单版本范围、流程、验收、节奏与版本内接口决策 | `docs/plans/*.md` |
| 仓库协作与同步规则 | `AGENTS.md` |

不要把同一规则复制到多个文件中长期维护。

## 4. 工作规则

### 4.1 版本优先
- 开发工作必须围绕一个明确版本进行
- 如果当前任务不属于任何已定义版本，先更新 `docs/prd.md` 和对应 plan，再开始实现
- 对尚未进入开发的版本，不提前编写完整 plan；在准备启动该版本时再创建
- 不允许把后续版本范围偷偷塞进当前版本

### 4.2 文档同步
- 修改产品边界或版本目标：更新 `docs/prd.md`
- 修改当前版本的接口决策或数据结构：先更新对应 `docs/plans/*.md`
- 当前稳定接口契约或 API 通用规范发生变化：同步更新 `docs/api-spec.md`
- 修改技术结构或工程约束：更新 `docs/architecture.md`
- 修改当前版本流程、验收或交付内容：更新对应 `docs/plans/*.md`

如果一次代码改动影响多个维度，必须在同一个变更中同步更新相关文档。

### 4.3 先计划后实现
- 新功能、较大重构或跨文件改动，应先确认对应版本 plan 已存在
- 如果 plan 不存在或明显过期，先更新计划文档
- 只有在计划和边界清晰后，才进入代码实现

## 5. 工程底线

- 保持实现与 `docs/architecture.md` 的目录规范一致
- 保持接口行为与 `docs/api-spec.md` 一致
- 不在 UI 组件中写复杂业务规则
- 不跳过输入校验、错误码设计和基本测试
- 不引入与当前版本无关的基础设施复杂度

## 6. 变更检查清单

提交前至少检查：
- 当前改动属于哪个版本
- 是否改动了产品范围、接口契约或架构约束
- 是否需要同步更新相关文档
- 是否满足当前版本计划中的验收标准
- 是否运行了与本次改动相关的验证命令

## 7. 验证要求

- 文档改动：至少检查文件结构、命名和引用路径正确
- 代码改动：至少运行与改动直接相关的测试、类型检查、构建或手动验证
- `packages/web` 的质量检查统一运行 `pnpm check`，该命令必须包含 TypeScript 类型检查；自动修复统一运行 `pnpm fix`
- 不要新增绕过 `pnpm check` / `pnpm fix` 的零散 lint、format 或 typecheck 脚本
- 不要在未验证的情况下宣称“完成”或“已通过”

如果仓库尚未建立完整脚本，必须明确说明未能运行哪些验证。

## 8. Git Commit 规范

### 8.1 提交格式

采用 `<type>(<scope>): <subject>` 格式，例如：

```
feat(api): 支持自然语言创建事件草稿
fix(web): 修复日期选择器时区偏移问题
docs(api-spec): 补充草稿确认接口错误码
refactor(services): 将解析逻辑抽离到 domain/parser
test(drafts): 补全缺失标题字段的边界用例
chore(deps): 升级 drizzle-orm 至 0.30.x
```

### 8.2 类型（type）

| 类型 | 含义 | 示例 |
|------|------|------|
| `feat` | 新功能 | 新增 API 路由、页面组件 |
| `fix` | Bug 修复 | 修复解析错误、UI 显示异常 |
| `docs` | 仅文档变动 | 更新 `prd.md`、`api-spec.md`、README |
| `style` | 不影响代码逻辑的格式变动 | Oxfmt、Rslint 自动格式化 |
| `refactor` | 重构（既不是 feat 也不是 fix） | 函数提取、目录迁移、重命名 |
| `perf` | 性能优化 | 减少重复查询、优化 bundle 体积 |
| `test` | 测试相关 | 新增/修改单元测试、e2e 测试 |
| `chore` | 构建/工具/依赖变动 | 升级依赖、调整 CI 配置、pnpm workspace 变更 |
| `ci` | 持续集成配置变动 | GitHub Actions、部署脚本 |
| `revert` | 回滚提交 | 回滚某次有问题的 commit |

### 8.3 作用域（scope）

作用域为可选项，用于指明改动的主要模块。推荐值：

- `api` — Nitro API routes、服务层
- `web` — Vite React 前端页面与组件
- `mobile` — Capacitor 移动端壳体
- `schemas` — `packages/schemas` 共享 Zod schema
- `shared` — `packages/shared` 共享类型与工具
- `services` — 领域服务层
- `db` / `orm` — 数据库、迁移、ORM 配置
- `parser` — 时间/文本解析相关
- `deps` — 依赖升级（可省略具体模块）
- `*` — 跨多个模块的大范围改动

### 8.4 主题（subject）

- 使用祈使句，描述**做了什么**而非做过了什么
- 首字母小写，句尾不加句号
- 控制在 50 个字符以内
- 必须能独立理解，不依赖上下文

正确示例：
- `feat(api): 支持 POST /drafts 自然语言解析`
- `fix(web): 修正 AllDay 事件跨天显示逻辑`

错误示例：
- ~~`feat(api): Added draft parse API`~~（使用过去时、英文与项目语言不一致）
- ~~`fix(web): bugfix.`~~（无信息量、带句号）

### 8.5 正文与脚注（可选）

当一次提交需要说明背景、影响面或关联 issue 时，在主题后空一行书写正文：

```
feat(api): 接入 chrono-node 做基础时间解析

- 用 chrono-node 替换手写正则，覆盖 ISO、相对时间等常见格式
- 服务层保留中文规则引擎作为 fallback
- 暂不处理农历和节假日别名

Closes #12
```

正文规则：
- 每行不超过 72 个字符
- 使用 `-` 列表说明具体改动点
- 说明**为什么**做这次改动，而不仅是**做了什么**
- 需要关联 issue/PR 时，使用 `Closes #xxx`、`Refs #xxx`

### 8.6 多模块改动的处理

- **优先拆分成多个独立提交**：文档更新、代码实现、测试补充分别提交
- 若必须合并提交（如重构涉及多包联动），在正文中分模块说明
- 禁止将无关改动（如格式化、功能代码、文档）混入同一个 `feat` 或 `fix` 提交

## 9. PR（Pull Request）提交规范

### 9.1 基本原则

- 请基于 PR 添加新功能。
- 每个 PR 只做一件事：只实现或修改单一功能；鼓励尽可能小、粒度尽可能细的 PR。
- 大功能应拆分为多个独立 PR 分步提交。

### 9.2 标题与描述

PR 标题与描述需清晰完整，内容包含：

- **标题**：一句话说明本 PR 新增/修改了什么。
- **功能描述**：说明该功能的作用与使用方式。
- **实现思路**：简要说明技术选型或核心实现逻辑。
- **测试方式**：如何验证该功能正常运行。

### 9.3 合并要求

PR 合并后，主分支代码需保持可运行状态，任意时间查看应能复现演示效果。

## 10. 禁止事项

- 未经文档更新直接改变版本范围
- 在没有对应 plan 的情况下展开大功能开发
- 让 `prd.md`、`architecture.md`、`api-spec.md` 重复维护同一细节
- 引入超出当前版本目标的功能膨胀
- 用临时实现绕开已定义的接口契约
